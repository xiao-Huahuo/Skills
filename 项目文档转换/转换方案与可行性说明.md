# 项目文档转换方案与可行性说明

## 1. 当前目录情况

`项目文档转换/` 下目前存在两套一一对应的文件：

### `original/`
- `MetaWeave(元织)-任务书.docx`
- `MetaWeave(元织)-开发计划书.doc`
- `开发规范.md`
- `测试计划和报告.md`
- `需求规格说明书.md`
- `项目启动通知.docx`

### `latter/`
- `任务书.docx`
- `开发计划书.docx`
- `开发规范.md`
- `测试计划和报告.md`
- `需求规格说明书.md`
- `项目启动通知.docx`

经检查，`latter/` 中现有文件与 `original/` 中对应文件哈希一致，说明 `latter/` 当前是原文档的完整拷贝，适合作为后续改写目标目录。

补充确认：

- `latter/` 中的开发计划书已由旧版 `.doc` 转为 `.docx`
- `latter/` 中的任务书文件名已调整为 `任务书.docx`

这意味着后续 `latter/` 目录中的 Word 文档都可以统一按 `.docx` 的精细改写流程处理。

## 2. 总体可行性判断

结论：**可行，但需要分类型处理，且应采用“只改文字、不改结构”的保守策略。**

本次任务的目标不是重新写一套新文档，而是在尽量保留以下内容的前提下，将文档内容从原项目 `MetaWeave（元织）` 转向当前项目 **云枢观策（CloudPolicy）**：

- 原有版式
- 原有标题层级
- 原有表格结构
- 原有段落数量与相对篇幅
- 原有措辞密度与正式文风
- 原有封面、页眉页脚、编号、目录、页码等文档外观元素

只要采用正确流程，这个任务对当前目录中的文档是可执行的。

---

## 3. 不同文件类型的可行性与处理方式

### 3.1 Markdown 文档：可直接安全改写

适用文件：

- `开发规范.md`
- `测试计划和报告.md`
- `需求规格说明书.md`

处理方式：

- 保留原始 Markdown 结构
- 保留标题级别、表格列数、列表层级
- 按段落改写内容，使主题从 `MetaWeave（个人知识中枢）` 转向 `云枢观策（民生政策理解、公共服务提效与基层信息可达性的智能化服务平台）`
- 优先保持段落体量接近，不做无必要扩写或缩写

风险较低，可控性最高。

### 3.2 `.docx` 文档：可保格式改写，但必须按底层 XML 精细处理

适用文件：

- `任务书.docx`
- `开发计划书.docx`
- `项目启动通知.docx`

处理方式：

- 解包 `.docx`
- 仅修改 Word XML 中的文字节点
- 尽量不重建段落、不重排 run、不改样式定义
- 修改完成后重新打包并校验文件完整性

这种方式通常可以保留：

- 字体、字号、加粗、颜色
- 段落样式和缩进
- 页眉页脚、页码、目录、表格、图片
- 大部分局部字符格式

主要风险：

- 同一句话可能被拆分为多个 run
- 文本可能出现在表格、文本框、页眉页脚、批注或域代码中
- 如果粗暴整段替换，可能破坏局部格式

因此 `.docx` 的改写应采用“精准替换文字节点”的方式，不能采用整段覆盖式修改。

### 3.3 当前 Word 处理结论

由于 `latter/` 中原开发计划书已经转为 `.docx`，后续目标目录内不再存在必须先转换的旧版 `.doc` 文件。

因此当前执行口径调整为：

1. `latter/` 下所有 Word 文档统一按 `.docx` 精细改写
2. `original/` 中旧版 `.doc` 仅作为参考来源，不作为直接编辑对象
3. 后续主要格式风险集中在复杂表格、页眉页脚、文本框和 run 拆分，而不再是 `.doc` 转换本身

## 4. 内容改写的核心原则

后续正式执行时，应严格遵循以下原则：

### 4.1 保持“文档骨架”不变

不主动改变：

- 标题层级
- 章节顺序
- 表格结构
- 编号体系
- 页眉页脚
- 目录、封面、签章占位、日期格式样式

### 4.2 只将“项目语义”整体迁移

要把原文档中围绕 `MetaWeave` 的项目背景、目标、功能、技术架构、测试对象、实施计划，整体迁移为围绕 **云枢观策** 的描述。

当前项目的核心定位，应以仓库现有材料为准，主要依据包括：

- `README.md`
- `doc/README-NOW.md`
- 代码实际结构与已存在功能

### 4.3 尽量保持句子体量接近

不是简单做名词替换，而是：

- 保持句子长短接近
- 保持段落数量接近
- 保持同类章节的信息密度接近
- 对表格项尽量做“同槽位替换”

### 4.4 优先吸收 README 中已经存在的真实内容

对于功能列举、技术栈、项目亮点、系统结构、部署方式、业务闭环、展示能力等章节，**优先使用当前项目 `README.md` 与 `doc/README-NOW.md` 中已经明确写出的真实内容进行充实**，而不是仅做空泛的句式改写。

执行原则如下：

- 如果 README 已经明确写出真实模块或能力，应直接吸收并重写进目标文档
- 如果 README 对某一部分描述较充分，允许该部分在新文档中比原文更充实
- 对“功能清单、技术栈、亮点展示、业务流程、页面结构、数据流向”等内容，不以机械保持句长为最高优先级，而以真实反映当前项目为最高优先级
- 只有在 README 及现有说明文档没有覆盖该信息时，才再去查代码或其他文件补足

### 4.5 不编造与当前项目明显不符的功能

如果原文中存在当前项目没有的能力，应处理为以下三类之一：

- 替换成当前项目已有且最接近的能力
- 下调为“规划中/扩展项”，前提是整份文档允许这样表达
- 删除该能力对应的具体承诺，但保留段落体量与结构稳定性

禁止把原项目特性原样照搬到当前项目文档中，导致文档与代码现实不一致。

### 4.6 禁止“换皮式改写”，必须做“同位置重写”

本任务不是把 `MetaWeave` 的表述简单替换为 `CloudPolicy` 的名称，也不是保留原项目叙事框架后仅做场景词替换。后续执行时，必须警惕以下错误：

- 仍沿用 `MetaWeave` 的底层产品叙事，只把项目名替换为 `CloudPolicy`
- 把 `CloudPolicy` 误写为“个人知识库”“文件管理进化体”“知识网络编织器”等与当前项目定位不符的产品
- 将当前项目的核心价值错误落在“整理个人知识资产”或“组织个人文件网络”上

正确做法是：在保留原段落位置、章节骨架、篇幅约束和正式文风的前提下，对每一段进行**同位置重写**。也就是说，要重建该段的问题定义、服务对象、价值落点与功能语义，使其真正符合 `CloudPolicy/云枢观策` 的实际定位，而不是对原段落做换名词式改写。

其中尤其要注意：

- `CloudPolicy` 的核心叙事应落在政策理解、公共服务提效、基层信息可达性、智能检索、文档解析、风险审计、智能问答等真实能力上
- 不得把“个人知识管理”当作默认母叙事再向政务场景生硬迁移
- 若发现改写结果在深层逻辑上仍然是原项目，只是表层词汇被替换，应视为改写失败，必须整段重写

### 4.7 继承原句式与逻辑骨架，但以 README 事实做破坏式改写

改写时不仅要保持原文的章节位置、段落功能和大致论述推进顺序，还应尽量继承原句式结构与论证逻辑。也就是说，原文先提出什么问题、再给出什么定位、最后把价值落到什么层面，新文也应在相近位置完成同类动作，而不是完全脱离原文重新自由发挥。

但这种“继承”只限于表达骨架，**不等于保留原项目内容内核**。实际写作时，应以当前项目 `README.md` 与 `doc/README-NOW.md` 中已经确认的真实定位、真实模块和真实能力为基础，对原文进行**破坏式改写**：

- 保留原句式骨架和逻辑推进
- 主动打碎原项目的产品假设、问题定义、用户对象和价值落点
- 用当前项目的真实能力重新填充同一位置
- 必要时允许整句甚至整段在语义层面彻底重建，只保留其在文档中的结构职能

这里的“破坏式改写”指的是：如果原段落虽然结构合适，但其底层产品观、场景设定或价值主张属于 `MetaWeave`，就必须把这些内容拆掉，换成 `CloudPolicy/云枢观策` 的真实叙事，而不是在旧叙事上做修补。

执行时应优先满足以下顺序：

1. 当前 README 与现有代码事实
2. 原段落的结构职能、句式骨架与逻辑推进
3. 原段落的篇幅和信息密度接近

如果三者冲突，优先保证“当前项目事实正确”，其次保证“同位置重写成立”，最后才是“表面句长接近”。

### 4.8 允许围绕当前项目做增删，禁止为保原文而保原文

后续改写不应机械追求“每一句都保留、每个句式都复刻”。在同位置重写成立的前提下，允许根据 `CloudPolicy/云枢观策` 的真实项目内容，对原文进行有方向的增删：

- 可以补入当前项目中真实存在、且对该段表达有帮助的新内容
- 可以删除原文中对当前项目无意义、无对应物或会带偏叙事的句子
- 可以替换掉虽然结构相似、但已经明显不适配当前项目的旧句式

这里的判断标准不是“是否像原文”，而是：

- 是否更准确地反映当前项目
- 是否仍然保留原段在整份文档中的结构职能
- 是否让改写后的段落读起来像为当前项目而写，而不是由旧项目勉强修补而来

因此，允许出现以下情况：

- 原段落略微扩充，以纳入当前 README 中更关键的真实能力
- 原段落局部收缩，以删除当前项目完全用不上的旧背景、旧比喻和旧价值表达
- 原句式被保留一部分，但其中若干分句被替换、重排或直接删除

总之，改写目标是“为当前项目写出成立的正式文档”，而不是“最大程度保留旧项目句子”。

### 4.9 允许打破原分点结构，主动突出当前项目亮点

在“推荐方案”“系统方案”“总体设计”“优势特色”“技术路线”等章节中，不必机械沿用原文的分点数量、分组方式和说明顺序。如果原文结构会削弱当前项目的真实亮点，允许主动调整表达重心，必要时可以增点、减点、并点或重排。

后续改写时，应优先突出 `CloudPolicy/云枢观策` 当前更有辨识度的真实能力，例如：

- Agent 决策与可解释推理轨迹
- 可视图谱与结构化结果展示
- 展示入口、业务系统、智能体、管理后台联动的一体化平台
- PDF / Word / Excel / 图片 / 截图 / 扫描件统一接入的多模态处理链
- 多端共用的响应式 UI 与统一视觉系统
- 发布、审核、公开展示、评价反馈、追踪分析组成的闭环能力

这类章节的改写目标，不是“忠实保留原方案的讲法”，而是“借用原章节的位置，写出当前项目真正值得被强调的方案亮点”。

因此允许出现以下处理：

- 原来是四点，改写后变成三点或五点
- 原来偏底层技术分层，改写后转为“平台亮点 + 核心机制 + 部署方式”
- 原来强调存储、监控、文件关系，改写后改为强调 Agent、中台能力、可视化结果和业务闭环

只要章节职能仍然成立，且内容更贴近当前项目现实，就不必为了形式对齐而压制亮点表达。

## 5. 建议的执行顺序

### 第一阶段：建立映射基线

需要完成：

- 汇总 `MetaWeave` 与 `云枢观策` 的项目定位差异
- 提炼当前项目已实现能力、可宣称能力、不可宣称能力
- 为六份文档建立统一术语替换表与功能映射表
- 以 `README.md` 与 `doc/README-NOW.md` 为主源，整理可直接纳入文档的真实功能、技术栈、页面结构与项目亮点

### 第二阶段：先处理 Markdown

建议先改：

1. `需求规格说明书.md`
2. `测试计划和报告.md`
3. `开发规范.md`

原因：

- 可见性高
- 修改成本低
- 能先稳定项目术语、模块命名与技术叙事
- 便于反向指导后续 Word 文档的改写口径

### 第三阶段：处理 `.docx`

建议顺序：

1. `项目启动通知.docx`
2. `任务书.docx`
3. `开发计划书.docx`

原因：

- `项目启动通知.docx` 文件较小，适合作为保格式改写流程验证样本
- 验证通过后再处理体量更大、格式更复杂的任务书与开发计划书

## 6. 后续实际执行时需要完成的具体工作

### A. 项目事实基线整理

- 从 `README.md`、`doc/README-NOW.md`、相关实现代码中提炼当前项目真实能力
- 形成统一项目描述口径：项目名称、定位、目标用户、核心模块、技术栈、部署方式、测试环境
- 对 README 中已写明的功能亮点、技术架构、系统页面、交互流程、展示能力进行结构化归纳，作为后续文档主材料

### B. 建立统一映射表

至少要建立以下映射：

- 项目名称映射
- 项目愿景与口号映射
- 用户角色映射
- 核心功能模块映射
- 技术栈映射
- 风险项映射
- 测试对象映射
- 交付成果映射

### C. Markdown 文档改写

- 在 `latter/` 中直接修改 `.md`
- 保留原结构，只替换内容
- 改完后检查章节连续性、编号、表格对齐和术语一致性

### D. Word 文档保格式改写

- 对 `latter/` 中的 `.docx` 解包编辑
- 仅替换文字节点
- 处理正文、表格、页眉页脚、可能存在的文本框
- 重新打包并验证能正常打开

### E. 统一校对

- 交叉检查六份文档的项目名称、年份、角色、功能、模块命名是否一致
- 检查是否仍残留 `MetaWeave`、个人知识中枢、文件管理进化等旧项目表述
- 检查文档描述是否与当前项目实际代码能力冲突
- 检查关键亮点是否已把 README 中的重要真实内容写进去，而不是只保留抽象描述

## 7. 主要风险与应对

### 风险 1：Word 文档局部格式丢失

应对：

- 不整段覆盖
- 尽量按文字节点替换
- 先用小文件验证流程，再处理大文件

### 风险 2：旧项目功能与当前项目不匹配

应对：

- 以当前仓库事实为准
- 不做机械替换
- 对无法映射的条目做同级别重写，而非硬套

### 风险 3：多文档术语不一致

应对：

- 开工前先建立统一映射表
- 每处理一份文档后同步回看统一术语

### 风险 4：README 信息丰富度与原文结构之间的张力

应对：

- 在“总体骨架不变”的前提下，允许功能、亮点、技术栈等章节适度扩充
- 对高信息密度章节优先保证内容真实完整，不机械追求逐句等长
- 扩充时仍保持版式稳定，不随意新增无关章节或打乱原编号体系

---

## 8. 结论

这个任务**可以做**，而且可以在较高程度上实现“保留原格式、仅迁移文字内容”的目标。

但正确做法不是一次性粗暴全量替换，而是：

1. 先建立当前项目事实基线与统一映射表
2. 先处理 `Markdown`
3. 充分吸收 `README.md` 与 `doc/README-NOW.md` 的真实内容
4. 再处理小型 `.docx`
5. 再处理大型 `.docx`
6. 最终做全套一致性校对

后续正式开始执行时，应只在 `latter/` 中修改，不触碰 `original/`。
