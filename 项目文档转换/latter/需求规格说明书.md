# MetaWeave（元织）软件需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档是为 **“MetaWeave（元织）”** 软件产品编写的需求规格说明书。MetaWeave 旨在从底层文件系统进化，打造一款个人知识中枢，以解决数字化办公时代知识资产碎片化、检索困难、信息孤岛等核心痛点。

本报告的核心目的在于：
1. **明确需求：** 详尽、准确地定义 MetaWeave 软件产品的各项功能需求、非功能需求以及外部接口，为后续的设计、开发和测试工作提供依据。
2. **统一认识：** 确保项目所有相关人员（包括客户、开发者、测试人员、管理者等）对产品的目标和功能有清晰、一致的理解。
3. **界定范围：** 明确本项目的开发边界，描述软件产品将实现的功能以及不涉及的范围。
4. **管理风险：** 识别项目潜在风险，为项目规划和管理提供决策支持。

通过本报告，我们将对 MetaWeave 进行精确的定义，指导整个开发过程，确保最终交付的产品能够实现“缝合碎片，编织智慧”的愿景。

### 1.2 项目背景

- **项目名称：** MetaWeave（元织）个人知识中枢
- **愿景：** 缝合碎片，编织智慧。
- **痛点背景：** 数字化办公导致资料散乱（信息孤岛）、附件不可搜（内容黑盒）、隐私审计缺失（被动容器）及整理成本高。
- **解决方案：** 通过原生集成 OCR、流式解析、RAG（检索增强生成）及 Agent 智能体技术，实现全量知识的透明化与智能化管理。

### 1.3 定义与缩写

| 术语/缩写      | 全称及定义                                                   |
| :------------- | :----------------------------------------------------------- |
| **RAG**        | Retrieval-Augmented Generation（检索增强生成）：通过检索私有知识库内容来增强大模型输出的准确性。 |
| **Agent**      | 智能体：具备感知环境、自主决策并调用工具（如 OCR、搜索）执行复杂任务的 AI 逻辑单元。 |
| **OCR**        | Optical Character Recognition（光学字符识别）：将图片或扫描件中的图像文字转化为计算机可编辑文本的技术。 |
| **向量数据库** | 专门用于存储和检索高维向量数据的数据库（如 ChromaDB），用于实现语义相似度匹配。 |
| **CoT**        | Chain-of-Thought（思维链）：展示 AI 推理过程的技术，使用户能够直观看到智能体处理问题的逻辑。 |
| **DFA**        | 确定有穷自动机 (Deterministic Finite Automaton)，用于高效敏感词过滤。 |
| **OT**         | 操作转换算法 (Operational Transformation)，用于多用户协同编辑的冲突处理。 |

**优先级定义：**
- **极高 (P0)：** 核心价值所在，**第一版必须包含**，缺失则产品无法发布或目标完全无法达成。
- **高 (P1)：** 非常重要，应在第一版或紧随其后的第一个迭代版本中包含。
- **中 (P2)：** 有价值，但可推迟到后续版本，或可有替代方案。
- **低 (P3)：** 锦上添花，资源允许时实现。

## 2. 任务概述

### 2.1 目标

MetaWeave 是一款从底层文件管理进化而来的个人知识中枢。其开发目标是：

- **利益：** 为用户提供一种全新的知识管理体验，从繁琐的手动整理中解放出来，实现知识的自动关联、智能检索和主动服务，最终提升个人和团队的知识产出效率。
- **目标：**
    1. **实现全格式内容感知：** 通过 OCR 和流式解析，使 PDF、图片、代码等非文本文件不再是“内容黑盒”。
    2. **构建智能知识大脑：** 利用 RAG 和 Agent 技术，自动将碎片化文件编织成语义关联的知识图谱。
    3. **提供主动式服务：** 从静态记录跃迁至动态审计，实现风险预警、重要日期提醒等智能自驱动功能。
    4. **打造安全可控的数据平台：** 内置隐私脱敏、多端同步、存证导出等功能，确保用户数据的安全性与合规性。

产品范围覆盖从个人知识消费者、管理者到团队协作伙伴的核心需求，但不包括针对特定行业（如医疗、金融）的垂直解决方案。

### 2.2 用户类与特性

| 用户类                    | 特性与期望                                           | 使用频率 | 专业水平 |
| :------------------------ | :--------------------------------------------------- | :------- | :------- |
| **普通用户 (知识消费者)** | 侧重全量搜索、智能摘要、白话转换等“消费型”需求。     | 高频     | 初级     |
| **知识管理者 (核心用户)** | 侧重图谱交互、双链维护、存证导出等“管理型”需求。     | 高频     | 高级     |
| **协作伙伴 (团队成员)**   | 侧重共享协同、版本追溯、冲突解决等“协作型”需求。     | 中频     | 中级     |
| **系统管理员 (运维人员)** | 侧重隐私脱敏、数据备份、异步任务监控等“运维型”需求。 | 低频     | 专家     |

### 2.3 假定与前提

- **假定：**
    - 用户已具备基础的计算机操作能力。
    - 用户运行客户端的设备具有稳定的网络连接（用于云端同步和 AI 服务）。
    - 用户承诺提供的文件不包含恶意代码。
- **前提：**
    - 系统依赖的 LLM API 服务（或本地模型服务）可用且性能稳定。
    - 运行环境具备足够存储空间以支撑用户知识库增长。

### 2.4 系统架构简述

- **客户端：** 基于 Vue 3 + Vite 的 Web 端，及通过 Electron 打包的跨平台桌面应用。
- **服务器：** 基于 FastAPI 的异步应用服务器，结合 SQLite/PostgreSQL（关系型）与 ChromaDB（向量型）双引擎。
- **异步层：** 利用 Redis + Celery 处理耗时的 OCR、LLM 推理及 Embedding 任务。

## 3. 外部接口需求

### 3.1 用户界面

- **风格：** 采用现代化、简洁、类似 GitHub Desktop 的设计风格，以文件管理为核心，辅以智能交互面板。严格遵循《开发规范.md》中关于前端 UI 和配色的规定。
- **布局：** 主界面分为三栏：
    - 左侧：文件树 / 功能导航
    - 中间：内容预览区 / 文件列表
    - 右侧：智能助手面板（知识图谱、对话、推荐）
- **主要组件：**
    - 文件/文件夹拖拽上传区域
    - 支持标签、收藏、分享的右键上下文菜单
    - 全局搜索栏，支持自然语言和前缀查询
    - 文件列表视图（列表、网格）
    - 文档阅读器（支持 Markdown、PDF、图片等）
    - 星空图谱交互区，支持节点拖拽、缩放、点击查看详情
    - 对话式 AI 智能体窗口
- **快捷键：** 支持常用快捷键，如 `Ctrl/Cmd + K` 快速搜索，`Ctrl/Cmd + S` 保存。
- **消息提示：** 统一采用顶部滑出式通知，用于提醒同步状态、任务完成、重要日期等。

### 3.2 硬件接口

- **本地文件系统：** 通过 Python `watchdog` 库监听本地文件夹的创建、修改、删除事件，实现文件的实时同步。

### 3.3 软件接口

| 接口名称                  | 目的                                     | 通讯方式              | 数据格式 | 备注                                                         |
| :------------------------ | :--------------------------------------- | :-------------------- | :------- | :----------------------------------------------------------- |
| **大语言模型 (LLM) 服务** | 智能总结、白话翻译、代码分析、智能体对话 | HTTP/HTTPS (RESTful)  | JSON     | 支持 OpenAI API 格式，支持多模型路由（Kimi、GPT-4o、DeepSeek） |
| **对象存储服务**          | 存储用户上传的原始文件                   | S3 兼容 API           | 二进制流 | 支持 MinIO、阿里云 OSS                                       |
| **关系型数据库**          | 存储用户信息、文件元数据、链接关系       | SQLModel (SQLAlchemy) | —        | PostgreSQL 13+ 或 SQLite 3.35+                               |
| **向量数据库**            | 存储文档向量，支持语义检索               | HTTP                  | JSON     | ChromaDB，支持 HNSW 索引                                     |

### 3.4 通信接口

- **协议：** 使用 HTTPS 保证数据传输安全（TLS 1.2+）。
- **WebSocket：**
    - **目的：** 用于服务端主动推送消息，如文件同步状态更新、任务进度、实时协作时的用户在线状态。
    - **重连机制：** 实现自动重连（指数退避算法），客户端本地缓存待同步队列。
- **邮件/SMTP：**
    - **目的：** 发送重要日期提醒、密码重置等通知。

## 4. 需求规定

### 4.1 功能性需求

本节将系统功能分为“智能化增强功能”与“基本操作与自动化功能”两大模块进行详述。

#### 4.1.1 智能化增强功能 (Intelligence & Automation)

此模块是系统的核心竞争力，利用大模型与向量检索技术，让软件具备自动理解、关联和审计的能力。

##### 深度感知与检索

| ID         | 功能名称              | 功能描述                                                     | 优先级      | 技术实现参考                                                 |
| :--------- | :-------------------- | :----------------------------------------------------------- | :---------- | :----------------------------------------------------------- |
| **[F-01]** | **语义意图搜索**      | 支持自然语言查询，通过内容含义找回相关资料。例如，输入“关于合同风险的文档”，系统通过嵌入模型匹配相关文件，而非依赖精确文件名。支持基于 Trie 树的前缀查询。 | **高 (P1)** | **后端**: Sentence-Transformers 或 OpenAI Embedding；**算法**: 余弦相似度、Trie Tree |
| **[F-02]** | **交互式关系图谱**    | 以星空地图形式可视化展示全库文件的语义关联与引用链路，支持缩放、拖拽、点击查看详情等交互。 | **高 (P1)** | **前端**: ECharts / D3.js (力导向图)；**后端**: 向量余弦相似度算法计算文件关联 |
| **[F-03]** | **知识双向链接**      | 在文档之间手动或自动建立前因后果的逻辑丝线。系统应根据内容自动建议链接（如检测到引用关系时弹出建议）。 | **中 (P2)** | **后端**: 正则表达式提取 `[[文件名]]` 模式；**逻辑**: 在关系数据库中建立 `links` 关联表 |
| **[F-04]** | **图片文字识别(OCR)** | 自动识别并提取图片、扫描件或截图中的文字内容，存入检索索引。支持常见图片格式，并优化噪声去除。 | **高 (P1)** | **库**: PaddleOCR / Tesseract；**架构**: FastAPI + Celery 异步任务队列处理 |
| **[F-05]** | **任务资料聚合**      | 根据当前任务上下文（如用户选中的关键词、打开的文档），自动聚合散落在各处的相关碎片化资料。 | **高 (P1)** | **架构**: RAG (检索增强生成)；**后端**: 基于上下文关键词的语义向量召回 |

| **[F-13]** | **代码逻辑漏洞分析**   | 针对存入的代码文件或代码块，系统自动进行静态分析，检查逻辑错误并给出优化建议。 | **低 (P3)** | **库**: Python `ast` 模块；**AI**: 针对代码优化的微调模型 API |
